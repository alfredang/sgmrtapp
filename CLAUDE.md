## Project

SGMRT is a native iOS 17+ SwiftUI app (Swift 6) for planning trips across Singapore's MRT
network: shortest-route estimation between stations, a route breakdown by line, a bundled
MRT map PDF, and optional live LTA DataMall context (service alerts + crowd density).

## Build system

The `.xcodeproj` is **generated by XcodeGen from `project.yml`** — never edit
`SGMRT.xcodeproj/project.pbxproj` or `SGMRTApp/Info.plist` keys directly; change `project.yml`
and regenerate. `Info.plist` is also generated from the `info.properties` block in `project.yml`.

```bash
xcodegen generate                       # after any project.yml change

# Build for simulator
xcodebuild -project SGMRT.xcodeproj -scheme SGMRT \
  -destination 'platform=iOS Simulator,name=iPhone 17 Pro Max' build

# Build + install on a physical device (see ios-install-device skill)
xcodebuild -project SGMRT.xcodeproj -scheme SGMRT -configuration Debug \
  -destination 'id=<HARDWARE_UDID>' -allowProvisioningUpdates -derivedDataPath build
xcrun devicectl device install app --device <UDID> \
  build/Build/Products/Debug-iphoneos/SGMRT.app
```

There are no unit tests or lint config in this repo.

## Configuration / secrets

- `LTA_ACCOUNT_KEY` flows: `Config/Secrets.xcconfig` (gitignored) → `Config/Local.xcconfig`
  (committed, optionally includes Secrets) → `LTAAccountKey` Info.plist key → read at runtime
  by `LTADataMallClient`. The app runs fully without a key; only live LTA data is skipped.
- App Store Connect credentials live in a gitignored `.env` (template `.env.example`); the
  `.p8` key stays outside the repo. See `SUBMISSION.md` and `scripts/asc_submit.py`.

## Architecture

The data flow is `MRTNetwork` (graph) → `RoutePlanner` (search) → `JourneyPlannerViewModel`
(`@MainActor ObservableObject`, the single source of truth) → SwiftUI views.

- **`Services/MRTNetwork.swift`** builds the network graph in its initializer from hardcoded
  per-line station-name lists. Station IDs are derived as `name.lowercased()` with spaces →
  hyphens, so a station appearing on multiple lines (e.g. interchanges) collapses to one node
  with a merged `lines` set. Edges are bidirectional `TrackEdge`s weighted in minutes (2 min
  per hop, 6 for the Changi Airport spur). **To add/fix a station or line, edit the relevant
  `builder.add(.line, [...])` list here** — there is no external data file.

- **`Services/RoutePlanner.swift`** is a Dijkstra search over `StateKey(stationID, line)` —
  state is keyed on *both* station and the line you arrived on, so line changes cost a
  `transferPenalty` (4 min). After the search it groups consecutive same-line edges into
  `RouteStep`s; `transferCount = steps.count - 1`.

- **`ViewModels/JourneyPlannerViewModel.swift`** owns `network`, `planner`, and the LTA client,
  plus all published UI state (`start`/`end`/`route`/`alerts`/`crowdDensity`). `calculateRoute()`
  is synchronous; `refreshLiveData()` is async and best-effort (LTA failures degrade to an error
  message, never block the route). Defaults start/end to Jurong East → Marina Bay.

- **`Services/LTADataMallClient.swift`** wraps two LTA DataMall OData endpoints
  (`TrainServiceAlerts`, `PCDRealTime`) with the `AccountKey` header. Crowd density is fetched
  per-line for the route's lines (capped at 3). No key → empty results, not an error.

- **Views** (`ContentView` tab host → `JourneyPlannerView`, `RouteSummaryView`, `LiveServiceView`,
  `StationPickerView`, `MapPDFView`) are thin and read from the shared view model.
  `Resources/SingaporeMRTMap.pdf` is the bundled map rendered by `MapPDFView`.

- **`Models/MRTModels.swift`** holds `Station`, `MRTLine` (raw values match LTA line codes used
  in `PCDRealTime` queries), `TrackEdge`, `RouteStep`, `RouteResult`, and the LTA response types.
